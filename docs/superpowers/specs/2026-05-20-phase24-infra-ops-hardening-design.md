## Stream A — PostgreSQL Client-Cert Auth

### Motivation

`DATABASE_URL` currently carries a plaintext password as a bootstrap secret. The WireGuard topology already provides network-level isolation; adding mTLS client certificates eliminates the password entirely from both the NUC prod instance and the WSL dev instance.

### Both instances in scope

| Instance | Host | Used by |
|---|---|---|
| NUC prod | `10.10.0.2:5432` over WireGuard | VPS backend container + scheduler container |
| WSL dev | `localhost:5432` (WSL-local docker-compose pg) | Local dev + `run-tests.sh` test container |

*[HIGH-3 applied; MED-P2-3 applied]* CI/test containers also exercise the cert auth path via an ephemeral CA + cert generated in container setup. The CI job configures the test PG with a server cert signed by the CA and connects with `sslmode=verify-full` — covering both client-auth and server-cert verification paths (see §Testing).

### Per-instance CA isolation

*[HIGH-P2-1 applied]* Each PG instance gets its own independent CA. A compromised dev cert must not authenticate to prod.

- **WSL dev CA:** generated by `scripts/pg-cert/generate-ca.sh`, stored at `~/.dashboard-pg-ca/dev-ca.key` (mode `0400`) and `~/.dashboard-pg-ca/dev-ca.crt`. Never copied to NUC. Used only to sign dev client certs and configure dev PG's `ssl_ca_file`.
- **NUC prod CA:** generated by `scripts/pg-cert/generate-ca.ps1`, stored at `C:\dashboard\pg-cert\ca.key` (DACL: trader user only) and `C:\dashboard\pg-cert\ca.crt`. Never transferred over the internet — generated on NUC directly or transferred via WireGuard SSH only. Used only for prod PG and the VPS backend client cert.
- The two CAs share no roots and are never cross-signed.

### Certificate layout

```
scripts/pg-cert/
  generate-ca.sh              # WSL: generate dev CA (stored at ~/.dashboard-pg-ca/)
  generate-client-cert.sh     # WSL: dashboard_backend client cert signed by dev CA
  install-dev.sh              # WSL: install certs into dev PG + pg_hba patch
  generate-ca.ps1             # NUC: generate prod CA (stored at C:\dashboard\pg-cert\)
  generate-client-cert.ps1    # NUC: client cert for VPS backend, signed by prod CA
  install-nuc.ps1             # NUC: install + pg_hba patch + DACL enforcement
docs/RUNBOOK-pg-cert-rotation.md  # Cert rotation + APP_SECRET_KEY rotation runbook
```

### `pg_hba.conf` change

```
# Before

# After
hostssl  dashboard  dashboard_user  10.10.0.0/24  cert  clientcert=verify-full
# ROLLBACK LINE (leave commented — remove only after one full phase-cycle confirmed):
# host  dashboard  dashboard_user  10.10.0.0/24  scram-sha-256
```

*[MED-4 applied]* The password line is commented (not deleted) for the Phase 24 cycle — operator rollback in < 2 min without a migration. Remove at Phase 25 start.

WSL dev `pg_hba.conf` equivalently switches the `127.0.0.1/32` line.

### Backend DSN

*[HIGH-2 applied]* SSL cert paths come exclusively from Docker-mounted volume + env vars. Reading from `app_secrets` is circular (DB connection required to decrypt → cert paths required to connect).

```
postgresql+asyncpg://dashboard_user@10.10.0.2:5432/dashboard
  ?sslmode=verify-full
  &sslcert=/run/secrets/pg_client.crt
  &sslkey=/run/secrets/pg_client.key
  &sslrootcert=/run/secrets/pg_ca.crt
```

Env vars: `PG_SSL_CERT_PATH`, `PG_SSL_KEY_PATH`, `PG_SSL_CA_PATH`. Cert auth activates when `PG_SSL_CERT_PATH` is set; password auth used otherwise (emergency rollback only). Both `backend` and `scheduler` containers receive these env vars — same client cert pair, different Docker secrets mount point documented in `docker-compose.yml`.

No `password=` field in the DSN when cert auth is active.

### Threat model and APP_SECRET_KEY handling

*[HIGH-1 applied; HIGH-P2-4 applied]* After dropping the PG password, `APP_SECRET_KEY` (Fernet master key in `.env`) becomes the sole bootstrap secret.

- **Risk:** Root on VPS + `.env` read → full `app_secrets` decryption.
- **Accepted risk:** Root on VPS is already game-over. `.env` protected at OS level.
- **Mitigations required this phase:**
  1. `scripts/deploy.sh` asserts `stat -c '%a' .env` == `600`. Deploy fails if wrong.
  2. `docs/RUNBOOK-pg-cert-rotation.md` §"APP_SECRET_KEY rotation" documents the full 6-step procedure:
     1. Generate new key.
     2. Set `APP_SECRET_KEY_OLD={current}` and `APP_SECRET_KEY={new}` in **both** `backend` and `scheduler` container env.
     3. Restart both containers (now decrypt with either key, encrypt with new key via `MultiFernet([new, old])`).
     4. Run `scripts/reencrypt-app-secrets.py` — iterates every `app_secrets` row per namespace, decrypts with old key, re-encrypts with new key, in a single transaction per namespace. Interrupted run is safe: resume the script; `MultiFernet` continues to serve both keys.
     5. Remove `APP_SECRET_KEY_OLD` from env in both containers, restart both.
     6. Verify by reading every namespace via `GET /api/admin/secrets`.
  3. Moving `APP_SECRET_KEY` to Docker secrets / host keyring is a post-v1.0 hardening item.

### Validation

- `psql` from WSL using client cert, no password → connects.
- `psql` from WSL using wrong cert → `FATAL: certificate authentication failed`.
- `psql` from WSL with no cert → `FATAL: certificate authentication failed`.
- Backend health endpoint returns 200 after cert switch on both NUC and WSL.
- Scheduler container connects to PG via cert auth (same client cert) on startup.
- CI matrix job: see §Testing.

### Rollback procedure

*[MED-4 applied]* If cert auth wedges prod:
1. Operator SSHes to NUC: uncomment `scram-sha-256` line in `pg_hba.conf`, run `pg_ctl reload`.
2. Unset `PG_SSL_CERT_PATH` in `.env` on VPS → both containers fall back to password DSN.
3. Run `scripts/restart-backend.sh` (restarts backend + bounces nginx).
4. Time to recovery: < 2 minutes. No migration required.

## Stream B — Multi-Worker Uvicorn

### Motivation

The backend runs as a single uvicorn worker. Before scaling to N workers, all long-running tasks that must not run N× must be extracted to a dedicated single-process scheduler container.

*[CRIT-1 applied]* The nonce, replay cache, and commission buffer are **already Redis-backed** from Phases 11d/5c — no migration needed:
- `orders_service.py:276,1013,1022,1496` — nonce: `redis.set(... nx=True)` + `GETDEL`
- `orders_service.py:1301,1682` — modify replay: `_modify_replay_lookup/_store` are Redis-backed
- `order_event_consumer.py:79–98,426,847` — commission buffer is Redis-backed

Stream B is entirely about the scheduler container split, the task inventory, and `_last_position_tick_at`.

### Sub-division: Chunk B1 before Chunk B2

*[MED-2 applied]* Multi-worker is enabled only AFTER B1 lands and runs in prod for ≥24 h at N=1.

### B1 — Long-running task inventory and scheduler container split

*[CRIT-P2-2 applied]* Every `asyncio.create_task` / `.start()` / `.run()` call in `app/main.py` lifespan must be triaged. Categorisation rule:

- **Both containers** — idempotent pub/sub listeners that populate per-process in-memory cache; fine to run in both.
- **Scheduler container only** — outbound to shared external resource (WS, gRPC streams), wall-clock cron jobs, or in-process state with no external truth.
- **API workers only** — nothing currently fits (all non-trivial state is either Redis/DB-backed or belongs in the scheduler).

### BrokerRegistry ownership

*[CRIT-P2-1 applied]* `BrokerRegistry` is used in two ways:
1. **Order dispatch** — `app/api/orders.py:24,70,138,207,226,263,327,472,631,705` inject it via `get_broker_registry()` to call `registry.get_client(gateway_label)`.
2. **Health probing + discovery** — `main.py:347–358` runs `health_probe_loop()` and feeds `BrokerDiscoverer`.

**Pattern chosen: independent instances in both containers.**

- **API workers:** construct a `BrokerRegistry` for outbound order dispatch only. Do NOT run `health_probe_loop()`. Broker health state is read from DB/Redis (written by the scheduler container's health probe loop).
- **Scheduler container:** constructs a `BrokerRegistry` that runs `health_probe_loop()`, owns `BrokerDiscoverer`, owns `OrderEventConsumer`. The health probe writes health state to DB/Redis so API workers can read it.
- Multiple gRPC clients to the same sidecar are supported (verified Phase 4 design).
- `get_broker_registry()` FastAPI dependency returns the per-process `BrokerRegistry` instance; this works in both containers since both construct their own.

`main.py` lifespan change: keep `BrokerRegistry` construction + gRPC client setup; remove `health_probe_loop()` start. The `health_probe_loop()` start moves to `app/scheduler.py`.

### quote_engine and coinbase

*[CRIT-P2-2 applied]* `quote_engine.start()` opens WebSocket streams to quote sources (IBKR, Schwab, etc.) and writes ticks to `bars_1s`. Under `--workers 4` without extraction: 4× upstream WS connections + 4× writes to the hypertable. `_coinbase_task` has the same problem. Both move to the scheduler container (see task inventory above).

`bar_service.start()` consumes the quote engine's output and is the sole writer to `bars_1s` — it also moves to the scheduler container.

### Scheduler container

```yaml
# docker-compose.yml (after B1)
services:
  backend:
    command: uvicorn app.main:app --host 0.0.0.0 --port 8000
             --workers ${UVICORN_WORKERS:-1}
    environment:
      - PG_SSL_CERT_PATH=/run/secrets/pg_client.crt
      - POSTGRES_POOL_SIZE=${POSTGRES_POOL_SIZE:-5}

  scheduler:
    command: python -m app.scheduler
    stop_grace_period: 360s          # MED-P2-1: allow long jobs to finish
    environment:
      - PG_SSL_CERT_PATH=/run/secrets/pg_client.crt
      - POSTGRES_POOL_SIZE=${POSTGRES_POOL_SIZE_SCHEDULER:-10}
```

*[LOW-P2-1 applied]* Entry point is `app/scheduler.py` (package-level `__main__` pattern via `-m app.scheduler`). The file is named `scheduler.py` within the `app/` package — no collision with stdlib `sched` since the project uses fully qualified imports. If the scheduler grows, convert to `app/scheduler/__init__.py` package later.

`app/scheduler.py` responsibilities:
- Constructs `BrokerRegistry` (dispatch-capable; runs `health_probe_loop()`).
- Starts all scheduler-container tasks from the inventory above.
- Starts APScheduler with all jobs (including Phase-23 CGT jobs after B1 migration).
- Writes heartbeat key `scheduler:heartbeat` to Redis every 30 s.
- No HTTP server.

*[MED-P2-1 applied]* Graceful shutdown:
- SIGTERM handler sets a shutdown event.
- APScheduler `shutdown(wait=True)`, max wait `SCHEDULER_SHUTDOWN_TIMEOUT_S` (default 300 s).
- `stop_grace_period: 360s` in `docker-compose.yml` gives Docker enough time.

*[MED-P2-5 applied]* Redis key prefix: no global prefix convention exists in this codebase (verified — bare keys like `nonce:order:`, `mkt:mid:` are standard). Use `scheduler:heartbeat` — consistent with existing bare-key style. If dev and prod Redis ever share an instance, add env-suffix at that time.

`app/main.py` changes:
- Remove all scheduler-container tasks from lifespan (per inventory above).
- Keep `BrokerRegistry` construction and gRPC client setup; remove `health_probe_loop()` start.
- Keep all "Both" pubsub listeners.
- `UVICORN_WORKERS` env var (default 1 in B1, raised to 4 in B2).

### PG connection budget

*[HIGH-P2-3 applied]* Each container has an independent asyncpg pool. At N=4 API workers + 1 scheduler:

| Container | Processes | Pool size | Max overflow | Peak connections |
|---|---|---|---|---|
| `backend` (N=4) | 4 | `POSTGRES_POOL_SIZE` (default 5) | 5 | 4 × 10 = 40 |
| `scheduler` (N=1) | 1 | `POSTGRES_POOL_SIZE_SCHEDULER` (default 10) | 5 | 1 × 15 = 15 |
| **Total** | | | | **≤ 55** |

PG18 `max_connections` must be ≥ 55 + headroom for `psql` sessions + CI. Default PG18 `max_connections=100` is sufficient with these defaults. Document in `docs/RUNBOOK-pg-cert-rotation.md` §"Connection budget". Operator must verify `SHOW max_connections` before raising `UVICORN_WORKERS` beyond 4.

## Stream C — `account_balances` Table Decoupling

### Motivation

`broker_accounts.last_nlv`, `last_nlv_currency`, `last_nlv_at` are read by 5+ services. A dedicated current-state table isolates balance state from account metadata and enables cash-by-currency, buying-power, and margin columns in future phases.

### New table: `account_balances`

*[MED-1 applied]* `cash_by_currency` is JSONB for display-only read paths. ADR: per-currency rows (`account_cash_balances(account_id, currency)`) are the correct eventual shape for per-currency `FOR UPDATE` and margin-by-currency queries. JSONB is acceptable for Phase 24 because no margin-by-currency consumers exist yet and the column is uninstrumented. Adding per-currency queries against this JSONB column is the trigger to add the sibling table.

```sql
CREATE TABLE account_balances (
    account_id       UUID PRIMARY KEY REFERENCES broker_accounts(id) ON DELETE CASCADE,
    nlv              NUMERIC(20, 8),
    nlv_currency     VARCHAR(8),
    -- display-only; do NOT add per-currency queries — add account_cash_balances table instead
    cash_by_currency JSONB DEFAULT '{}',
    buying_power     NUMERIC(20, 8),
    margin_used      NUMERIC(20, 8),
    updated_at       TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

### Expand-contract migration

*[CRIT-4 applied]* Live read paths in `orders_service.py:1925,1944` and `position_sizing_service.py:85,239,246,248` prevent single-phase column drop.

**Alembic 0074 (expand — this phase):**

2. Backfill: `INSERT INTO account_balances (account_id, nlv, nlv_currency, updated_at) SELECT id, last_nlv, last_nlv_currency, last_nlv_at FROM broker_accounts WHERE last_nlv IS NOT NULL`.
3. `BrokerDiscoverer` dual-writes atomically (see §Dual-write atomicity below).
4. All 5+ read sites redirected to `AccountBalanceService.get_current()`.
5. `orders_service.py:1940` comment block updated to reference `AccountBalanceService`.
6. `broker_accounts.last_nlv*` columns remain — no drop.

- Pre-commit grep hook enforces this on every commit after Chunk C ships.

*[MED-P2-2 applied]* Grep path is `backend/app/` only — explicitly excludes `backend/tests/`, `backend/alembic/versions/`, and `backend/app/models/` (ORM model must keep the column definition until the drop migration runs).

### Dual-write atomicity

*[HIGH-P2-5 applied]* The two writes must be atomic — if the legacy write succeeds and the `account_balances` upsert fails, readers see stale NLV data.

`AccountBalanceService.upsert()` wraps both writes in a single SQLAlchemy transaction:

```python
async with session.begin():
    await session.execute(
        text("""
            UPDATE broker_accounts
            SET last_nlv = :nlv, last_nlv_currency = :currency, last_nlv_at = :at
            WHERE id = :account_id
        """), {...}
    )
    await session.execute(
        text("""
            INSERT INTO account_balances (account_id, nlv, nlv_currency, updated_at)
            VALUES (:account_id, :nlv, :currency, :at)
            ON CONFLICT (account_id) DO UPDATE
              SET nlv = EXCLUDED.nlv,
                  nlv_currency = EXCLUDED.nlv_currency,
                  updated_at = EXCLUDED.updated_at
        """), {...}
    )
```

Both writes commit together or both roll back. `test_dual_write_atomicity.py`: inject a post-legacy-write exception, verify `account_balances` row is not written.

### New service: `AccountBalanceService`

```python
class AccountBalanceService:
    async def get_current(self, account_id: UUID) -> AccountBalanceRow | None: ...
    async def upsert(self, account_id: UUID, nlv: Decimal, currency: str,
                     buying_power: Decimal | None = None,
                     margin_used: Decimal | None = None) -> None: ...
```

### FE impact

`AccountResponse` already exposes `nlv` / `nlv_currency`. The service layer change is transparent to the FE wire shape.
