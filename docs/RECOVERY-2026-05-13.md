## Issue 1 — VPS backend lifespan crash (502 on admin page)

**Symptom:** `https://dashboard.kiusinghung.com/api/admin/config` returns
502 (or 302 redirect to CF login from unauth'd test).

**Root cause:** VPS backend container is in `Restarting (3)` loop. Lifespan
crashes at `app/main.py:134`:

```
app.services.config.SecretDecryptError: fernet decryption failed for
ai.litellm_master_key; verify APP_SECRET_KEY/PREV_KEY rotation state
```

The 1 row in prod `app_secrets` (`ai.litellm_master_key`, the bootstrap
placeholder) was Fernet-encrypted with an `APP_SECRET_KEY` that doesn't
match what's in the VPS's `/home/trader/trading-dashboard/.env` right
now. The lifespan fail-CLOSEDs (security review M1 pattern), preventing
the backend from booting → admin API unreachable → can't rotate the
bad row via the documented `PUT /api/admin/secrets/...` flow.

**Cascade:** `bar_aggregator` + `schwab-sidecar` also restarting (likely
depend on backend); `nginx` reports unhealthy (upstream health probe).

### Recovery — Option A (recommended): delete the bad row

This is the clean path. The row is a known bootstrap placeholder
(`sk-bootstrap-rotate-me`); deleting it makes the lifespan re-mint a
fresh one encrypted with the current `APP_SECRET_KEY`.

```bash
# From a host that can reach the NUC PG. The password is read from the
# checked-out .env so it never appears in shell history or this doc.
PROD_PG_PASSWORD="$(grep '^DATABASE_URL=' /home/joseph/dashboard/.env \
    | sed -nE 's|.*://[^:]+:([^@]+)@.*|\1|p')"
docker compose exec -T \
  -e PGPASSWORD="$PROD_PG_PASSWORD" \
  test_postgres \
  psql -h 10.10.0.2 -U trader -d dashboard \
  -c "DELETE FROM app_secrets WHERE namespace = 'ai' AND key = 'litellm_master_key';"
```

Then bounce the VPS backend:

```bash
ssh -p 2222 trader@88.208.197.219 \
  'cd trading-dashboard && docker compose restart backend && docker compose restart nginx'
```

Per memory `nginx_backend_recreate_502.md`, the nginx restart is required
or the upstream stays stale and the 502 persists.

Verify recovery:

```bash
ssh -p 2222 trader@88.208.197.219 \
  'cd trading-dashboard && docker compose ps backend'

# Should show: Up (healthy), not Restarting
```

### Recovery — Option B (if Option A insufficient)

The `APP_SECRET_KEY` mismatch suggests rotation happened without
populating `APP_SECRET_KEY_PREV` for dual-key Fernet support.

1. Check VPS `.env`: ensure `APP_SECRET_KEY` and (if rotated)
   `APP_SECRET_KEY_PREV` are both present, with the previous value in
   `_PREV` so existing ciphertext can be migrated.
2. If you remember the previous value, set it as `APP_SECRET_KEY_PREV`
   before the backend restart.
3. Otherwise Option A is the only path — the placeholder row is
   cheap to recreate.

### Aftermath — also affects local backend

The same row blocks the local backend container (running in WSL on the
NUC) if it ever restarts cleanly. Today's local instance got past this
because it was already running. After fixing prod, also run:

```bash
docker compose restart backend  # local, from /home/joseph/dashboard
```
